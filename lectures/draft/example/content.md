# 用 Vibe Coding 與 Codex 打造可重複使用的講義網站
> 從一次性的 AI 產出，走向可維護、可擴充、可持續迭代的內容工作流。

[summary]
- 主軸 **從草稿到網站** | 把口語化教學內容整理成可維護的 Markdown 講義系統
- 核心 **模板與 Skill** | 用固定規格降低 AI 輸出的隨機性，讓結果更穩定
- 實作 **終端機與外掛** | 比較 Codex CLI 與編輯器外掛兩種操作方式
- 延伸 **團隊協作** | 用 Commit、PR、Worktree、測試把 Vibe Coding 變成長期可用的工程流程
[/summary]

## 為什麼要把講義做成網站sadasd

### 從 PPT 轉向可維護的內容系統
- 課程主題是 AI 寫程式與 Agent Skill，講義本身也應該反映這種工作方式
- 與其用傳統 PPT，不如直接做成網頁，讓版面、互動和輸出形式更自由
- 網頁可以在電腦、平板、手機瀏覽，也能再延伸成 PDF 或靜態部署

> **真正的價值不在第一次生成**
> AI 很擅長快速做出「第一版」，但一份講義要能長期使用，重點是後續修改是否輕鬆、結構是否穩定。

### 為什麼要封裝成 Agent Skill
- 如果每次都從零提示 AI，產出風格與品質都會飄
- 把模板、規格與腳本封裝成 Skill，之後遇到新主題就能重複使用
- 內容編輯回到 Markdown，最終輸出仍維持網站形式

## 在終端機安裝與使用 Codex

### 準備工作區
- 在編輯器中先開一個乾淨資料夾，提供 AI Agent 明確的工作範圍
- 不論使用 VSCode、Cursor 或 Antigravity，都可以先把資料夾開好
- 本次示範主要透過終端機操作，編輯器只是拿來看檔案變更

### 確認 Node.js 與版本管理

[flow]
1. 打開終端機，執行 `node -v`，確認本機已有 Node.js
2. 若尚未安裝，先安裝 NVM，讓不同 Node 版本可切換與更新
3. 執行 `nvm install --lts` 安裝長期支援版
4. 再次執行 `node -v` 驗證安裝結果
[/flow]

```prompt [label="Mac 安裝 NVM"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

```prompt [label="Windows NVM"]
https://github.com/coreybutler/nvm-windows/releases
```

### 安裝 Codex

```prompt [label="安裝 Codex CLI"]
npm i -g @openai/codex
```

- 安裝後執行 `codex`
- 可使用 ChatGPT 帳號登入、裝置代碼登入或 API Key
- 如果是在無瀏覽器的環境，可用裝置代碼方式登入
- 使用 API Key 時要注意額度與成本控制

> **登入方式的選擇是環境問題，不是功能問題**
> 對多數開發者來說，直接登入 ChatGPT 帳號最省事；但 VM、遠端主機或無 UI 環境，就需要改用裝置代碼授權。

## 先讓 AI 產出第一版網站

### 初版 Prompt 的設計重點

[flow]
1. 明確定義目標：只產出單一 HTML，降低部署複雜度
2. 指定風格：例如日系風格、漸層、具有設計感的區塊
3. 指定結構：標題、標語、分層內容、社群連結、側邊目錄
4. 等待 AI 先完成可看的第一版，再進入優化
[/flow]

```prompt [label="生成教學講義網頁"]
請扮演熟悉 AI 主題授課的講師，以「Agent Skills」為主題設計講義
講義用純 html 網頁（CSS/JS 都放裡面）呈現
日系風格的設計感，部分文字與背景可以搭配適當的漸層
開頭會根據內容呈現標題，然後下面有發人深省的標語
會根據不同層級與意圖呈現對應的區塊（ex: 大小標題、陳述、列點、流程）
最後放上社群連結鼓勵大家訂閱（FB/YouTube/Medium）
側邊欄會有對應的目錄，方便跳轉到對應位置
```

### 初版可用，但不夠好維護
- 單純生成 HTML，短期看起來很快
- 但當你要調整段落、語氣、結構與細節時，很容易陷入痛苦
- 與 AI 反覆對話修改太隨機，直接改 HTML 又太低效

## 把一次性產出轉成模板

### 核心思路：內容與呈現分離
- 不要把知識內容直接綁死在 HTML
- 讓 AI 幫你設計一套 Markdown 規格，專門描述標題、列點、流程、提示框等結構
- 再用腳本把 Markdown 轉回穩定的 HTML 頁面

> **模板不是限制創意，而是保留掌控權**
> AI 可以自由幫你起草，但真正可維護的系統，一定要讓人類能快速精修內容。

### 逆向設計 Markdown 規格

```prompt [label="把 HTML 逆向拆成模板"]
我希望剛剛生成出來的網頁，是可以透過 Markdown 格式編輯內容的
請幫我將網頁結構拆解（列點、步驟、圖片、表格、註解）、分層次（大標、中標、小標），並以常見的 Markdown 語法來設計規則
最後建立一個腳本，我只要編輯 Makdown 文件，網頁的 HTML 就會及時改變
操作方式寫在 README.md 文件中，並建立 package.json
```

### 這段 Prompt 在做什麼
- 不是再叫 AI 生一次網站
- 而是要求 AI 幫你把輸出格式「產品化」
- 內容放在 Markdown，渲染邏輯放在腳本，操作方式放在 README

## 建立 Agent Skill 工作流

### 把流程封裝進 `.agents`
- Skill 的 `name` 用來定義能力名稱
- `description` 說明何時觸發、預期產出什麼
- `reference/` 放格式規範與範例
- `scripts/` 放 build、dev、generate-og 等腳本

### 建立 Skill 的 Prompt

```prompt [label="建立 course page skill"]
根據下面需求在「.agents」資料夾建立 Skill：
- 參考模板：根據剛剛腳本設計的規格，建立明確的 Markdown 使用規範
- 觸發時機：使用者提供初步想法想建立網頁時，會觸發這個 Skill
- 執行順序：這個 Skill 會先深入研究輸入的主題，再根據規範設計對應的 Markdown 文件；最後把 Markdown 文件透過腳本生成 html
- 特別提醒：預計會有多個主題，所以觸發時，請先建立新的資料夾，裡面存放新生成的 Markdown 與 HTML 檔案；並且建立獨立的 scripts 與 references 資料夾，確保可單獨執行
```

### Skill 的目標不是炫技
- 真正的目的，是把零散操作變成可複製流程
- 新主題進來時，AI 有依據可跟，不必每次重猜格式
- 這也是從玩具型 Vibe Coding，走向長期維護工具的重要一步

## 認識 Codex 外掛與介面

### 安裝編輯器外掛
- 在 Antigravity 的 Extensions 搜尋 `chatgpt`
- 找不到時，可用 `antigravity:extension/openai.chatgpt` 直接開啟安裝頁
- 安裝後打開任一檔案，點選上方 ChatGPT 圖示即可使用

### 終端機與外掛的差異
- 終端機操作較接近工程工作流，適合熟悉指令的使用者
- 外掛有更直觀的 UI，較適合初學者
- 兩者本質都是在和同一類型的 AI Agent 協作，差異主要在互動介面

## 從 0 到 1 之後，真正困難的是 1 到 100

### OpenSpec 與規格驅動開發
- AI 寫程式很快，但沒有規格文件的專案會越改越亂
- OpenSpec 讓你把白話需求整理成 proposal、design、specs 與 task
- 這能降低知識斷層，讓後續交接與迭代更有依據

```prompt [label="開始建立規格"]
請設計一個公司內部 Dashboard 系統，包含以下功能：
- 登入頁面（帳號密碼驗證，區分管理者與一般使用者）
- 首頁儀表板（顯示關鍵數據卡片：營收、訂單數、活躍用戶數、轉換率）
- 資料圖表頁（折線圖、長條圖，支援日期篩選）
- 員工管理頁（管理者可檢視、新增、編輯、刪除員工資料）

前端使用 React + TypeScript，使用 Mock API 模擬後端回應
這是初步需求，我們可以透過討論釐清細節後，參考 openspec 的 skill 執行
以最小可行性方案來規劃
```

### Skill 不只幫你生東西，也幫你維護秩序
- Commit Skill 讓多功能修改可以拆成可審核的提交
- PR Skill 讓變更描述更有結構
- Git Worktree 讓多個 AI Agent 能並行開發不同任務
- 測試 Skill 讓 AI 不只修當下錯誤，也能保護既有功能

> **人類最大的瓶頸不是寫得慢，而是審得慢**
> AI 大幅提升產出速度後，真正要補強的是規格、版本控制與測試，否則團隊只會更快產出更多難以維護的程式碼。

## 讓 Vibe Coding 走向產品化

### 導入測試的必要性

[flow]
1. AI 修 bug 可能順手改壞舊功能
2. 功能越多，人工回歸測試越不可能全面覆蓋
3. 測試讓你敢改、敢擴充，也能更快定位問題
[/flow]

### 用 AI 幫你建立測試案例
- 先讓 AI 列出測試情境，而不是直接產生測試碼
- 由人類審查情境是否完整，再讓 AI 實作
- 這比完全手寫更快，也比盲目全交給 AI 更可靠

## 結語

### 不要把 AI 成果當免洗餐具
- 真正有價值的不是「哪個工具最強」
- 而是你是否知道什麼時候該用哪種工具、怎麼把它們接成流程
- 當內容模板、Agent Skill、版本控制與測試能互相配合，AI 才會變成長期可用的生產力系統

> **最後帶走的一件事**
> AI 可以幫你加速第一版，但只有把流程設計好，你才有辦法把它變成可長期使用、可維護、可擴充的資產。
