# AI Agent：從聊天助手走向可執行的工作夥伴
> 真正改變工作的，不是模型多會回答，而是它能不能理解目標、使用工具、維持上下文，並在約束下把事情做完。

[summary]
- 定義 **AI Agent 不只是聊天** | 核心在於目標導向、工具調用、狀態管理與持續執行
- 實作 **Codex 與 Claude Code 的差異** | 一個偏向終端機與工作區操作，一個偏向編輯器與工程協作流程
- 設計 **不是把權限全開** | 好的 Agent 工作流要有邊界、回饋、審核點與失敗處理
- 落地 **先解決單一步驟痛點** | 從產生講義、整理 PR、建立測試開始，比從全自動主管助理開始更實際
[/summary]

## 先釐清：什麼才算 AI Agent

### 🧭 Agent 與一般聊天機器人的差別
- 一般聊天模型的重點是「回答問題」
- AI Agent 的重點是「朝目標推進」
- 它不只輸出文字，還會讀檔案、改程式、查資料、呼叫工具、根據結果再做下一步

### 🧱 AI Agent 的四個基本元件

[flow]
1. 目標 - 使用者交付的任務、限制與完成條件
2. 模型 - 負責理解指令、拆解任務、產生下一步決策
3. 工具 - 終端機、檔案系統、搜尋、API、瀏覽器、自訂函式
4. 狀態 - 暫存觀察、決策、上下文與中間成果
[/flow]

> **只會說，不算 Agent**
> 當模型只能提供建議，真正執行還是全靠人手時，它比較像顧問；當它能在明確邊界內自己採取行動，才開始接近 Agent。

## 為什麼 2026 的 AI Agent 特別值得學

### 📈 模型能力不只變強，而是變得更能工作
- 長上下文讓 Agent 更能理解整個 codebase 或長篇文件
- Tool use 讓模型不必只靠猜，可以自己查證、讀檔、執行命令
- 指令遵循與結構化輸出更穩，讓工作流可被腳本與系統接住

### ⚠️ 但更強不代表更能無人管理
- 工具一多，失誤成本就變高
- 有權限的 Agent 若沒有邊界，等同把高風險操作包裝成自然語言
- 導入 Agent 前，先想清楚哪件事值得自動化，遠比先選模型更重要

## OpenAI Codex 與 Claude Code：兩種典型工程 Agent 體驗

### 🛠️ OpenAI Codex 擅長的使用情境
- 在終端機中直接協作，靠讀 repo、跑命令、編修檔案完成任務
- 適合處理具體的工程工作，例如修 bug、補測試、整理文件、建置小工具
- 對於「看工作區後再動手」這種流程很自然，特別適合資深工程師協作

### 🧠 Claude Code 擅長的使用情境
- 更偏向在開發流程中扮演長時間思考與規劃助手
- 對大型重構、需求拆解、長鏈推理、風險分析常有不錯表現
- 在編輯器情境中，常被用來配合規格、Review 與架構推演

### 🔍 該怎麼理解兩者差異

[tags]
- [green] Codex：偏行動型，靠工作區與工具一步步把事做完
- [blue] Claude Code：偏規劃型，對複雜思路、設計討論與大面積變更較有優勢
- [orange] 共通點：都不是魔法，是否好用取決於任務定義、權限設計與上下文品質
[/tags]

> **不要問誰最強，要問誰更適合這一步**
> 真正成熟的團隊，不是把所有工作都丟給同一個 Agent，而是知道什麼任務該用哪種互動模式、哪種風險控管。

## 一個好用的 AI Agent，背後其實是工作流設計

### 🎯 先從任務拆解開始
- 模糊指令只會產生模糊結果
- 好的 Agent 任務通常包含目標、範圍、限制、輸出形式與驗證方式
- 若任務需要跨多步，最好明確指出哪些步驟可以自主，哪些步驟要先回報

```prompt [label="低品質指令"]
幫我把這個專案整理好。
```

```prompt [label="高品質指令"]
請先閱讀 repo，找出目前 build 失敗的原因。
修復問題後執行測試，若測試仍失敗請說明原因。
只修改與 build 問題直接相關的檔案，最後回報變更與驗證結果。
```

### 🗺️ 任務描述應至少交代五件事

[flow]
1. 目標 - 最後要完成什麼
2. 範圍 - 可以動哪些系統、檔案或模組
3. 限制 - 哪些不能做、哪些需要保守處理
4. 驗證 - 怎麼算完成，例如測試、build、畫面、輸出檔案
5. 回報 - 最後要怎麼說明結果
[/flow]

## 工具越多，Agent 越像同事，也越需要管理

### 🔧 常見工具類型
- 檔案系統：讀寫內容、產生檔案、調整設定
- 終端機：執行 build、test、lint、git、資料處理命令
- Web/搜尋：查文件、驗證最新資訊、比對官方規格
- Browser/UI：操作後台、點擊按鈕、擷取畫面
- 自訂函式：查資料庫、叫內部 API、送通知

### 🔐 工具權限設計的三個原則
- 預設最小權限，不要因為方便就讓 Agent 一開始什麼都能做
- 高風險動作要有人類覆核，例如刪資料、發信、轉帳、部署正式環境
- 讀取與執行權限要拆開，不要讓能看資料的 Agent 順便能無條件修改所有內容

> **Agent 最危險的不是答錯，而是做錯**
> 當模型只回錯一句話，通常還有人工補救；但如果它已經有權限直接改資料、送通知、推 commit，失誤的影響就會立刻放大。

## 用 Codex / Claude Code 實際落地時，最常見的四種工作流

### 1. Repo 讀懂與任務定位
- 先列出相關檔案
- 確認現有實作與使用者需求是否一致
- 找出問題發生點，而不是一開始就大改

### 2. 小範圍修改與驗證
- 只改與需求直接相關的區塊
- 修改後立刻跑測試、build 或最小驗證
- 若驗證失敗，先縮小原因，再決定是否擴大修改

### 3. 規格與文件生成
- 把口語需求轉成 README、設計稿、技能文件、提案文件
- 這類任務很適合 Agent，因為成本低、回饋快、容易人工審核

### 4. 重複性流程自動化
- 產生 commit message
- 整理 PR description
- 生成測試案例
- 建立講義、範本頁、報表與固定格式內容

## Case Study：把「AI Agent」主題做成講義網站

### 📚 為什麼這是一個好練習
- 內容生成、格式結構、建置腳本、OG 圖、catalog 更新，都是典型多步任務
- 每一步都能驗證，不需要直接碰正式系統
- 很適合用來訓練 Agent 的工作流設計能力

### 🧪 這類任務怎麼交給 Agent

[flow]
1. 定義主題與受眾 - 例如面向工程師、PM 或企業內訓學員
2. 要求生成 content.md - 內容採可維護格式
3. 補上 config.yaml - 處理標題、SEO、Hero 與引言
4. 執行 build / generate-og - 驗證輸出物是否齊全
5. 啟動 dev server - 確認後續可持續編修
[/flow]

```prompt [label="課程生成指令"]
扮演一位擅長 openai-codex 及 claude-code 的專家，
設計「AI Agent」的講義並生成網頁。
完成後直接啟動。
```

## 不要把 Agent 當黑盒子，觀察它的思考痕跡

### 👀 你該關注哪些訊號
- 它先讀了哪些檔案
- 它怎麼縮小問題範圍
- 它在哪一步選擇驗證，而不是直接硬改
- 它是否知道哪些資訊可能已過期，需要即時查證

### 🧯 三種危險訊號
- 還沒理解 repo 就開始大改
- 遇到錯誤就嘗試用更多修改掩蓋問題
- 對最新資訊、版本、套件規格做過度自信的假設

> **好的 Agent 不是總能一次答對，而是能穩定地自我修正**
> 工程工作本來就充滿不確定性，所以你要觀察的是它怎麼找線索、怎麼驗證、怎麼收斂，而不是只看第一個答案漂不漂亮。

## 讓 Agent 可長期使用的關鍵：技能、規則與記憶

### 🧩 為什麼要建立 Skill
- 把特定工作流固定下來，避免每次重講一次
- 讓 Agent 知道何時該觸發、要讀哪些參考、最後要輸出什麼
- 對團隊來說，Skill 也是把隱性經驗顯性化的方式

### 📝 除了 Skill，還要有規則
- commit 要分段，不要一個需求塞十幾種修改
- 高風險命令要先確認
- 測試與 build 是完成定義的一部分，不是可有可無
- 對外部文件、最新規格、價格、法律資訊要先查證

### 🧠 記憶的價值不是記住一切
- 而是記住穩定可重用的偏好與流程
- 例如專案的 commit 風格、測試習慣、發版規則、報告格式
- 把可標準化的東西外部化，才能讓 Agent 每次進來都更快對齊

## 企業導入 AI Agent，最容易卡住的不是技術

### 🚧 常見落地阻力
- 任務定義太抽象，無法驗證是否成功
- 權限設計太粗暴，不是全關就是全開
- 團隊沒有 code review、測試與版本控制習慣，AI 只會把混亂放大
- 主管期待一步到位的全自動，忽略漸進式導入更實際

### ✅ 建議的導入順序

[flow]
1. 選一個低風險、高重複的任務
2. 先建立輸入格式、輸出格式與驗證方法
3. 再決定要不要給更多工具與權限
4. 讓人類保留最後決策權
5. 累積成功案例後，再往更複雜流程擴張
[/flow]

## 結語：真正的競爭力，不是會用一個 Agent

### 🎯 最後要帶走的三個觀念
- AI Agent 的本質是工作流，不是聊天介面
- OpenAI Codex 與 Claude Code 都很強，但強在不同階段的協作方式
- 長期價值來自可維護的模板、技能、驗證機制與權限治理

> **未來最值錢的人，不是最會下 Prompt 的人**
> 而是最懂得怎麼把目標、工具、規則、驗證與風險控管接成一套穩定工作流的人。
